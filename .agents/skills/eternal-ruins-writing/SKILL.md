# Eternal Ruins Writing

Use this skill for rules text, chapter prose, examples, tables, and reviews of Eternal Ruins drafts in this repo.

## Quick Start

1. Read `references/style-notes.md` before making a substantial edit or review.
2. Treat `AGENTS.md` in the repo root as binding. If the book style and repo rules conflict, follow the repo rules.
3. Review in this order: terminology, heading structure, rules clarity, tone, then sentence-level polish.

## Core Style

- Write in a calm, confident, slightly lyrical voice. The prose should feel atmospheric, but still operational.
- Prefer concrete description over abstract commentary. Name what the Wanderer, the Guide, the scene, or the rule is doing.
- Keep procedural text direct. A good rules paragraph usually follows this pattern: trigger, instruction, result.
- Use second person for procedures and player-facing guidance. Use third person when describing Wanderers, Guides, regions, details, demons, and other game elements as setting objects.
- Avoid jokey or overly modern phrasing. The book’s voice is warm, but it is not casual internet speech.

## Terminology

- Capitalize game terms when they are being used as defined concepts. This includes terms such as `Guide`, `Wanderer`, `Action Roll`, `Region Track`, `Risk Oracle`, `Question Oracle`, `Hope`, `Despair`, `Feature`, `Detail`, `Oddment`, and `Fragment`.
- Use the book’s vocabulary. Prefer `Wanderer` over generic player-character language, `Guide` over GM, `make camp` over rest scene, and `location`, `region`, `feature`, and `detail` in their specific mechanical senses.
- Keep category names singular when introducing the concept, and plural when describing sets or lists.

## Structure

- Follow heading hierarchy strictly. Do not skip levels.
- Write chapter and section bodies in short paragraphs. Shift into bullets or tables when the text becomes procedural.
- Introduce a rule before giving its outcomes or options.
- Keep examples tightly tied to the rule they clarify.
- If the rendered design already uppercases headings through CSS, keep Markdown headings in readable title case unless the file’s existing convention clearly differs.

## Review Checklist

- Does the section sound like book prose rather than blog prose or design notes?
- Are the defined terms capitalized consistently?
- Does each rules block clearly state when it applies, what to do, and what happens next?
- Are list items parallel in structure?
- Are headings real Markdown headings instead of bold pseudo-headings unless the local pattern requires otherwise?
- Have repo-local rules been applied: Oxford comma, no contractions, no em dashes, and proper heading hierarchy?

## Source Caveat

The current full PDF has a placeholder `SOLO RULES` chapter on page 143. For solo-writing work, infer tone and formatting from the rest of the book, especially the `BASIC RULES`, `FREEFORM SCENES`, `EXPLORATION SCENES`, `CAMPING SCENES`, `CHARACTER CREATION`, `REGIONS`, `DETAILS`, and creature chapters.
